## 前言

本规范是 **Hermes Agent 中文 Start Book** 系列的「宪法」。它的目的是确保整个文档体系具备**高度的结构完整性、一致性、美观性与可维护性**，让读者获得专业、舒适且生产级的阅读体验。

本规范完全体现用户偏好：
- **结构化**：清晰层级、完整 Checklist、详细模板
- **美观**：合理使用 Emoji、徽章、特征卡片、表格
- **实用**：大量可复制命令、真实模板、最佳实践
- **PDF 友好**：紧凑布局、清晰层级、避免过长代码块横向溢出
- **自我进化**：内置审查清单，支持 Hermes 自动维护

## 1. 设计哲学

### 核心价值观
1. **读者为中心** —— 让中文用户「一看就懂、拿来就能用」
2. **生产级质量** —— 像对待代码一样对待文档（可 Review、可测试、可迭代）
3. **一致性至上** —— 打开任意文档都应有「同一个作者、同一套标准」的感觉
4. **美观与紧凑并存** —— 现代审美（参考 Tailwind 设计语言），同时适合 PDF 打印
5. **自我迭代** —— 文档本身可被 Hermes 阅读、Review 并持续优化

### 写作原则
- **实用优先**：每个章节都应包含「可直接复制」的命令、Prompt、模板
- **结构完整**：每个专题文档都应包含前言、核心概念、设计原则、最佳实践、模板、故障排除、审查清单
- **深度适中**：聚焦「如何设计高质量 Skill/Memory/Plan」，而非泛泛而谈
- **友好专业**：语气温暖且权威，像「资深同事」在认真分享经验

## 2. Frontmatter 标准（必须位于文件最顶部）

```yaml
---
title: 文档标题（建议中文为主，英文副标题可选）
description: 简短的一句话描述（用于 SEO 和概览）
version: "v2.1"
date: 2026-04-10
author: Hermes Agent（基于用户偏好深度定制）
tags: [hermes, skill, memory, guide, production-ready]
---

# 正文标题（与 title 一致或更具体）
```

**必填字段**：`title`, `version`, `date`, `author`

## 3. 整体文档结构模板

**推荐结构**（根据文档类型可灵活调整）：

1. **封面区**（标题 + Logo + 版本 + 徽章）
2. **前言**（目标读者、核心价值、如何使用本书）
3. **目录**
4. **核心章节**（按重要性排序）
   - 概念与原理
   - **高质量设计原则**（这是用户最重视的部分）
   - 实战模板（大量可复制内容）
   - 最佳实践
   - 故障排除与常见问题
5. **审查清单**（Checklist）
6. **附录**（术语表、版本历史、相关资源）
7. **页脚**（Made with ❤️ by Hermes Agent）

## 4. 排版与视觉规范

### 徽章与 Emoji 使用
- 重要程度：`★★★★★`、`★★★★☆`
- 状态徽章：**已完成**、`进行中`、`待完善`
- 推荐徽章风格：
  - ✅ 完成
  - ⚠️ 注意
  - 🔥 强烈推荐
  - 📋 模板

### 标题层级
- `#`：一级标题（文档主标题）
- `##`：二级标题（主要章节）
- `###`：三级标题（小节）
- `####`：四级标题（具体规则或模板名）

### 代码与命令
- 命令示例必须使用 `bash` 语言标识
- 重要命令前后增加空行，提高可读性
- 长命令建议拆行（使用 `\`）
- 提供「一键复制」友好格式（避免过多嵌套）

### 表格
- 必须有表头
- 推荐列：功能 | 描述 | 推荐度 | 备注
- 使用 `|` 对齐，保持美观

### 警告/提示框
使用以下格式：
> **💡 最佳实践**：...
>
> **⚠️ 注意**：...
>
> **✅ 推荐**：...

## 5. 内容质量要求

### 「如何设计高质量 X」章节必须包含
- 设计原则（3-7 条带编号的核心原则）
- 好的示例 vs 坏的示例（对比表格）
- 具体模板（可直接使用的 YAML/Markdown）
- 审查清单（Checklist）
- 版本演进建议（如何持续迭代）

### Memory / Skill 专项要求
- **Skill**：必须包含 `trigger conditions`、`numbered steps`、`pitfalls`、`verification steps`
- **Memory**：区分 `user profile` 与 `personal notes`，强调「减少未来用户 steering」

---

## 6. PDF 导出优化规范

- 保持布局紧凑（避免过多空行）
- 代码块不要过长（建议控制在 80 字符内）
- 使用 `---` 做清晰分隔
- 标题层级清晰，确保打印后层级明显
- 推荐使用 Chrome 浏览器「打印 → 另存为 PDF」测试效果
- 避免使用过大图片（当前本系列主要使用 Logo）

## 7. 自我审查 Checklist（每次更新必须通过）

```markdown
### 📋 本文档审查清单

- [ ] Frontmatter 完整且符合规范
- [ ] 标题层级逻辑清晰、无跳级
- [ ] 包含「高质量设计原则」章节
- [ ] 提供至少 2 个可直接复制的模板
- [ ] 所有命令均测试有效（或注明版本）
- [ ] 使用统一的 Emoji 与徽章风格
- [ ] 符合 PDF 紧凑布局要求
- [ ] 交叉链接指向其他相关文档
- [ ] 版本号与日期更新
- [ ] 已遵循本 Style Guide
```

## 9. 版本管理规则

- 每次重大更新升级 `version`（v1.0 → v1.1 或 v2.0）
- 更新 `date` 字段
- 在文档底部增加「版本历史」小节（可选集中维护）
- 重大重构建议创建 `vX.Y-重大更新记录.md`
