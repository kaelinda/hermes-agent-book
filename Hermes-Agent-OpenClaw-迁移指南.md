## 前言

如果你已经用惯了 OpenClaw，那么切到 Hermes Agent 时最容易出现两种感觉：

1. “这不就是另一个 Agent CLI 吗？”
2. “功能更多了，但我是不是得全部重学？”

答案都是否定的。

Hermes Agent 最适合 OpenClaw 用户的迁移方式，不是“推翻重来”，而是：

**保留你已经熟悉的 Agent 使用习惯，再把它升级成一个带有长期记忆、可自我进化、可多代理协作、可定时自动运行的完整系统。**

这篇指南专门解决迁移中的真实问题：
- 你原来的提示习惯、项目习惯、上下文文件，如何平滑迁过来
- 你熟悉的“单会话 Agent”工作方式，如何升级为 Hermes 的“记忆 + Skill + Plan + Subagent + Cron”闭环
- 哪些东西应该直接照搬，哪些东西应该借迁移机会顺手重构

> **✅ 推荐**：把这篇文档理解为“迁移地图”，而不是“安装文档”。
>
> 安装和基础配置请配合《快速入门完全实战指南》一起使用。

## 1. 为什么 OpenClaw 用户值得迁移到 Hermes

OpenClaw 的优势通常体现在：上手快、命令行心智简单、适合直接拿来当编码/对话助手。

而 Hermes 的优势，在于它不是只优化“这一轮对话”，而是优化“你未来每一轮对话的质量与成本”。

### 你得到的不只是替代品，而是升级包

| 维度 | OpenClaw 常见体验 | Hermes Agent 升级点 | 推荐度 |
|------|-------------------|--------------------|--------|
| 日常提问 / 编码 | 能直接对话与执行任务 | 同样能做，而且工具面更完整 | ★★★★★ |
| 复杂任务拆解 | 依赖你手动描述步骤 | 有 `/plan`、`todo`、结构化计划机制 | ★★★★★ |
| 长期复用经验 | 主要依赖你自己记住 Prompt | 可把经验沉淀成 Skill，并持续 patch | ★★★★★ |
| 用户偏好继承 | 需要反复提醒 | 有 `USER` / `MEMORY` / `SOUL.md` 三层记忆 | ★★★★★ |
| 多角色协作 | 通常靠多窗口 / 多实例 | 有 Profiles、Subagent、Cron 形成体系 | ★★★★★ |
| 自动巡检 / 定时任务 | 通常要外部脚本拼接 | 原生 `cronjob` 支持自包含任务 | ★★★★☆ |
| 长期演进 | 容易变成“每次重新教一遍” | 自带自我进化闭环 | ★★★★★ |

### 最值得迁移的 4 个原因

1. **你不用再一遍遍重复自己的偏好**  
   把沟通风格写进 `SOUL.md`，把稳定偏好记进 `USER`，把环境约定记进 `MEMORY`，后面所有任务都更顺。

2. **你能把高频套路沉淀成 Skill**  
   以前你可能保存了一堆 Prompt 片段；迁到 Hermes 后，可以把这些片段升级成真正可维护的 Skill。

3. **复杂任务终于可以先规划再执行**  
   对文档改造、代码迁移、批量检查、仓库治理这类任务，`/plan` 会明显提升稳定性。

4. **你能把“一个 Agent”变成“一个 Agent 系统”**  
   Profiles、Subagent、Cron 让你从单兵作战，升级到多角色分工和自动化协作。

## 2. OpenClaw → Hermes 概念映射表

迁移最怕的不是不会用，而是脑子里没有“对应关系”。先把概念映射清楚，迁移就会顺很多。

### 核心映射

| OpenClaw 里的习惯 | 在 Hermes 里的对应能力 | 迁移建议 |
|-------------------|------------------------|----------|
| 常驻 Prompt / 系统设定 | `SOUL.md` | 把你最稳定的语气、偏好、禁忌迁到 `~/.hermes/SOUL.md` |
| 常用提示模板 | Skill | 高频、可复用、带步骤的 Prompt 优先升级成 Skill |
| “记住我喜欢什么” | `USER` Memory | 把稳定偏好写入用户画像型记忆 |
| “记住这个项目怎么做” | `MEMORY` | 把环境、约定、工具坑位写入个人笔记型记忆 |
| 手工拆步骤 | `/plan` + `todo` | 复杂任务默认先规划 |
| 多开窗口分别做事 | Profiles / `delegate_task` | 长期隔离用 Profiles，临时并行用子代理 |
| 外部定时脚本 | `cronjob` | 周报、巡检、定时整理类任务直接迁入 Hermes |
| 一次性上下文说明 | `.context/` / 项目文档 | 稳定项目上下文优先文档化，而不是每次临时贴 |

### 心智模型变化

| 阶段 | OpenClaw 心智 | Hermes 心智 |
|------|---------------|-------------|
| 入门 | “我在用一个智能 CLI” | “我在训练一个长期合作的 Agent” |
| 进阶 | “我需要写更好的 Prompt” | “我需要设计更好的 Skill / Memory / Profile” |
| 高阶 | “我想批量自动化” | “我可以搭一个可持续进化的 Agent 工作流系统” |

### 典型工作流对比表

| 场景 | OpenClaw 常见做法 | Hermes 推荐做法 | 升级收益 |
|------|-------------------|----------------|----------|
| 临时问答 | 直接开聊，按轮修正 | 直接开聊，必要时让 Agent 读取项目上下文 | 基本零学习成本迁移 |
| 复杂改造任务 | 手工分步骤，边做边补充要求 | 先 `/plan`，再执行并做 verification | 成功率更高、返工更少 |
| 高频套路复用 | 保存 Prompt 到笔记或剪贴板 | 升级为 Skill 并持续 `patch` | 经验真正沉淀为资产 |
| 项目知识承接 | 靠聊天历史和个人记忆 | 写入 `.context/`、`MEMORY`、文档 | 换项目/换时间后也不容易失忆 |
| 多角色协作 | 多窗口、多实例、人工同步 | 长期用 Profiles，临时并行用 `delegate_task` | 隔离更清晰、切换成本更低 |
| 周期性巡检 | shell 脚本 + 人工触发 | `cronjob` + 自包含 Prompt | 自动化程度更高 |
| 偏好继承 | 每次重新提醒 | `SOUL.md` + `USER` + `MEMORY` | 长期合作体验明显更顺 |

> **💡 最佳实践**：迁移时不要试图 1:1 模拟 OpenClaw 的所有用法。
>
> 更好的做法是：先保留熟悉习惯，再逐步接入 Hermes 独有能力。

## 3. 迁移前检查清单

在动手之前，先明确哪些东西要搬，哪些东西不值得搬。

### 建议保留并迁移的内容

- 你长期稳定使用的系统风格 Prompt
- 你反复复制粘贴的高频任务模板
- 你常用的项目上下文说明
- 你已经验证有效的工作流习惯
- 你经常提醒 Agent 的偏好与禁忌

### 建议借机清理的内容

- 只用过一次的 Prompt 垃圾堆
- 大量语义重复的提示片段
- 已经过时的模型偏好说明
- 只为某个历史 bug 临时补丁的冗余提示
- 模糊、口语化、不可复用的操作说明

### 迁移前盘点表

| 类型 | 你是否已有 | 迁移目标 | 备注 |
|------|------------|----------|------|
| 人格 / 语气设定 | 是 / 否 | `SOUL.md` | 最先迁移 |
| 用户偏好 | 是 / 否 | `USER` | 只留长期有效信息 |
| 项目约定 | 是 / 否 | `MEMORY` 或 `.context/` | 别全塞进聊天里 |
| 高频 Prompt | 是 / 否 | Skill | 值得结构化 |
| 自动化脚本 | 是 / 否 | `cronjob` / 外部脚本 + Hermes | 看是否需要定时 |
| 多角色分工 | 是 / 否 | Profiles | 长期使用再拆 |

## 4. 15 分钟最小可用迁移路径

如果你想先快速跑起来，不要一上来就做全量迁移。先走“最小可用版本”。

### 第一步：安装并完成基础设置

```bash
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | sh
hermes setup
hermes model
hermes tools
```

如果你已经完成安装，请继续下一步。

### 第二步：先迁 `SOUL.md`

把你原来在 OpenClaw 里最稳定、最值钱的那段“人格 / 输出风格 / 工作方式”写到：

```bash
mkdir -p ~/.hermes
$EDITOR ~/.hermes/SOUL.md
```

优先写这 4 类内容：
- 你希望它的语气
- 你偏好的输出结构
- 你不希望它做的事
- 你最常做的任务类型

### 第三步：先跑一次真实任务

不要先折腾完所有配置再开始用。直接拿你最熟的一个任务测试：

```bash
hermes chat -q "请阅读当前仓库结构，先给出一个简洁计划，然后执行最小改动修复 README 中的过期链接。"
```

这一轮你主要观察 3 件事：
- 工具调用是否顺手
- 输出结构是否符合你习惯
- 是否明显减少你反复提醒的成本

### 第四步：把 1 个高频 Prompt 升级成 Skill

这是迁移分水岭。

如果你过去总在 OpenClaw 里反复输入类似：
- “先读代码，再列计划，再修改，再验证”
- “先看文档规范，再批量修复 Markdown”
- “先排查根因，再给出最小修复方案”

那它们就不该继续只是 Prompt，而应该变成 Skill。

### 第五步：再引入 Plan / Profile / Cron

顺序建议：
1. 先把单 Agent 用顺
2. 再使用 `/plan`
3. 再拆 Profiles
4. 最后接入 `cronjob`

> **⚠️ 注意**：不要第一天就把 Memory、Skills、Profiles、Cron、MCP 一次性全开。
>
> 这样只会让迁移体验变复杂，而不是丝滑。

## 5. 配置迁移：从“会用”升级到“顺手”

这一章讲的是：如何把 OpenClaw 的“提示习惯”，升级为 Hermes 的“长期配置资产”。

### 5.1 人格迁移：Prompt → SOUL.md

最值得先迁移的是人格与工作风格，而不是技巧性 Prompt。

#### 推荐结构

| 迁移内容 | 放到哪里 | 为什么 |
|----------|----------|--------|
| 沟通语气 | `SOUL.md` | 全局生效，最稳定 |
| 结构偏好 | `SOUL.md` | 比每次临时说更省心 |
| 禁忌事项 | `SOUL.md` / `USER` | 能减少反复纠偏 |
| 临时任务要求 | 当前对话 | 不要滥写入长期记忆 |

### 5.2 偏好迁移：反复提醒 → USER

适合放进 `USER` 的内容：
- 你喜欢中文还是中英混排
- 你偏好短回答还是完整方案
- 你是否喜欢表格、Checklist、代码模板
- 你常用的工作流、习惯、表达风格

不适合放进去的内容：
- 某次任务的临时要求
- 已完成事项
- 一次性上下文

### 5.3 项目经验迁移：聊天记录 → MEMORY / .context

Hermes 最大的差别，是你可以把“这个项目该怎么协作”系统化沉淀下来。

推荐拆分方式：

| 信息类型 | 推荐位置 |
|----------|----------|
| 稳定用户偏好 | `USER` |
| 环境事实、工具坑位、仓库约定 | `MEMORY` |
| 项目结构、架构、规范 | `.context/PROJECT.md`、`.context/ARCHITECTURE.md`、`.context/CONVENTIONS.md` |
| 需要重复执行的方法 | Skill |

### 5.4 提示模板迁移：Prompt 仓库 → Skill 体系

如果一个 Prompt 同时满足下面 3 条，就建议升级成 Skill：

1. 你会重复使用
2. 它有明确步骤
3. 它带有经验性判断或避坑逻辑

例如这些都非常适合迁移成 Skill：
- 仓库文档规范化
- PR 审查
- 系统性 Debug
- 每周项目巡检
- 技术调研总结

### 5.5 多实例习惯迁移：多窗口 → Profiles

如果你原先靠多个 OpenClaw 窗口扮演不同角色，那在 Hermes 里应该升级为 Profiles：

```bash
hermes profile create coder
hermes profile create researcher
hermes profile create assistant
```

之后可直接使用：

```bash
coder chat
researcher chat
assistant chat
```

这样每个角色都有独立记忆、技能、配置和会话，不会互相污染。

## 6. 工作流迁移：从单 Agent 到自我进化系统

这是迁移最值钱的部分。

### 6.1 日常编码工作流

#### OpenClaw 常见方式

- 直接提需求
- 不满意再追加说明
- 复杂任务靠你手动分步骤
- 好用 Prompt 复制到别处保存

#### Hermes 推荐方式

- 简单任务：直接执行
- 复杂任务：先 `/plan`
- 高频套路：沉淀 Skill
- 长期偏好：写进 `SOUL.md` / `USER`
- 项目事实：进入 `MEMORY` / `.context`

### 6.2 复杂任务工作流

下面是 OpenClaw 用户迁移后最该立刻采用的默认流程：

```text
收到复杂需求
→ 先让 Hermes 做 prerequisite checks
→ 输出 plan
→ 按 todo / 子任务推进
→ 完成后做 verification
→ 把可复用经验存成 Skill / Memory
```

### 6.3 并行工作流

如果你以前需要自己切多个窗口并来回同步信息，在 Hermes 里可以这样升级：

- 长期角色分工：Profiles
- 临时并行研究 / 审查：`delegate_task`
- 定时巡检 / 自动汇报：`cronjob`

### 6.4 迁移后的推荐成长路径

| 阶段 | 目标 | 重点能力 |
|------|------|----------|
| 第 1 天 | 跑通基础使用 | `SOUL.md`、单次真实任务 |
| 第 3 天 | 提升稳定性 | `/plan`、`USER`、`MEMORY` |
| 第 1 周 | 减少重复劳动 | Skill 设计 |
| 第 2 周 | 形成角色分工 | Profiles |
| 第 3 周 | 建立自动化闭环 | `cronjob`、周期性 Review |

## 7. 如何设计高质量迁移方案（核心重点）

很多迁移失败，不是因为工具不好，而是因为迁移策略错了。

### 高质量迁移的 8 大设计原则

1. **先迁人格，再迁技巧**  
   先把风格、结构偏好、禁忌写对，再考虑高级能力。

2. **先迁高频资产，再迁边缘资产**  
   先迁你每周都会用到的东西，而不是历史 Prompt 仓库。

3. **从单 Agent 开始，不要一口气多角色全开**  
   先让主流程稳定，再拆分 Profiles。

4. **把迁移当成清理机会，而不是无脑复制**  
   只迁高价值、长期有效、可复用的内容。

5. **Prompt 能升级成 Skill 的，就不要继续散落保存**  
   Hermes 的真正价值，在于把“用法”变成“资产”。

6. **项目事实应文档化，而不是长期靠聊天堆积**  
   `.context/`、README、专项文档，都会比聊天记录更稳定。

7. **复杂任务默认计划优先**  
   一旦任务会涉及多个文件、多轮修改、多方验证，就先 `/plan`。

8. **迁移不是终点，自我进化才是终点**  
   完成迁移后，要建立定期 Review、更新 Skill、清理 Memory 的习惯。

### 迁移成功的判断标准

| 判断项 | 说明 |
|--------|------|
| 你不再需要反复提醒语气和格式 | `SOUL.md` / `USER` 起作用了 |
| 高频任务开始出现稳定套路 | Skill 开始生效 |
| 复杂任务成功率提升 | `/plan` 和 verification 起作用了 |
| 多角色协作更清晰 | Profiles / Subagent 正常工作 |
| 你愿意继续沉淀资产 | Hermes 已从“工具”变成“系统” |

## 8. 好的迁移姿势 vs 坏的迁移姿势

| 维度 | 坏的迁移姿势 | 好的迁移姿势 |
|------|--------------|--------------|
| 心态 | 想 1:1 复制旧工具所有体验 | 以熟悉习惯为起点，接受能力升级 |
| 配置 | 先装一堆功能再说 | 先跑通最小闭环 |
| Prompt | 所有旧 Prompt 原样搬过来 | 只保留高频高价值部分 |
| 记忆 | 什么都往长期记忆里写 | 只写稳定、长期有效信息 |
| 技能 | 继续把套路存在备忘录 | 把可复用套路升级成 Skill |
| 多角色 | 第一时间拆 5 个 Profile | 先单 Agent，再逐步拆分 |
| 自动化 | 上来就排满 Cron | 先验证流程，再定时化 |

### 一个推荐的迁移节奏

#### 第 1 阶段：复刻熟悉体验
- 安装 Hermes
- 迁移 `SOUL.md`
- 跑 1 个真实任务

#### 第 2 阶段：引入结构化能力
- 用 `/plan` 处理复杂任务
- 写入第一批 `USER` / `MEMORY`
- 建立 `.context/`

#### 第 3 阶段：建立资产化能力
- 把高频 Prompt 改造成 Skill
- 拆出第一个 Profile
- 开始定期 Review

#### 第 4 阶段：建立自我进化闭环
- 周期性维护 Skill / Memory
- 用 `cronjob` 跑巡检或日报
- 持续迭代自己的 Agent 系统

### 3 个典型迁移案例

#### 案例 A：个人开发者

- **OpenClaw 旧习惯**：直接把需求扔给 Agent，复杂任务靠自己口头补步骤
- **迁移动作**：把代码风格和输出结构写进 `SOUL.md`；复杂任务默认先 `/plan`
- **迁移后收益**：减少“来回纠偏”，长任务的稳定性和可预测性明显提升

#### 案例 B：文档维护者 / 仓库 Owner

- **OpenClaw 旧习惯**：反复复制“先读规范、再修文档、再检查链接”的 Prompt
- **迁移动作**：把这类套路升级成 Skill，并将规范写进 `.context/` 或仓库文档
- **迁移后收益**：每次维护都更一致，适合长期做版本化迭代

#### 案例 C：研究 / 内容型用户

- **OpenClaw 旧习惯**：把调研、总结、写稿都混在同一会话里完成
- **迁移动作**：拆出 `researcher` Profile，临时资料比对交给 `delegate_task`
- **迁移后收益**：角色更清晰，调研型上下文不会污染编码或写作型 Agent

## 9. 团队迁移建议

如果不是你一个人迁，而是团队想从 OpenClaw 体系切到 Hermes，建议这样做：

### 团队推荐分层

| 层级 | 建议做法 | 目标 |
|------|----------|------|
| 个人层 | 每人维护自己的 `SOUL.md` / `USER` | 保留个人工作风格 |
| 项目层 | 建立 `.context/` 与统一文档规范 | 降低口口相传成本 |
| 角色层 | 用 Profiles 区分 coder / reviewer / researcher | 稳定职责边界 |
| 组织层 | 用共享 Prompt 经验沉淀 Skill 设计方法 | 形成团队知识资产 |

### 团队迁移时最常见的误区

1. 只迁工具，不迁方法论  
2. 每个人都各自乱存 Prompt，没有统一 Skill 规范  
3. 没有 `.context/`，导致项目知识仍然只存在聊天里  
4. 直接追求自动化，忽略了基础质量治理  

> **✅ 推荐**：先统一文档规范和 Skill 设计原则，再做规模化迁移。

## 10. 常见问题与故障排除

### Q1：我是不是要把 OpenClaw 的所有 Prompt 全搬过来？

不是。

建议只迁：
- 高频使用
- 结构稳定
- 真正提高质量
- 未来还会继续用

### Q2：我到底先学 Skill 还是先学 Memory？

先学 Memory 的边界，再学 Skill 的结构。

原因很简单：
- 你得先知道什么应该记住
- 再知道什么应该做成方法

### Q3：什么时候开始用 Profiles？

当你已经出现“不同任务风格互相污染”时，再开始拆。

例如：
- 研究型工作和编码型工作风格完全不同
- 工作 / 个人环境需要隔离
- 你想让某个 Agent 长期负责特定职责

### Q4：什么时候开始用 Cron？

当某个任务同时满足：
- 会重复发生
- 输入结构固定
- 输出目标明确
- 不需要每次人工强互动

例如：
- 每周仓库文档巡检
- 每日报告整理
- 周期性技能维护建议

### Q5：迁移后感觉“更强了，但也更复杂”，怎么办？

这很正常。

处理方式不是退回去，而是收敛范围：
1. 先只用单 Agent
2. 只保留 1 个 Skill
3. 只写最关键的 `SOUL.md`
4. 等主流程顺了，再继续加能力
