## 1. 什么是 Advanced Skills

**基础 Skill** 关注“完成一个任务”。

**Advanced Skills** 关注的是：
- 多任务之间如何协同
- 一个 Skill 如何在 3 个月后仍然可维护
- 如何让多个 Skill 组成一条稳定工作流
- 如何避免 Skill 数量增加后变得混乱、重复、失效

### 基础篇 vs 进阶篇

| 维度 | 基础 Skill 设计 | Advanced Skills 设计 |
|------|------------------|----------------------|
| 目标 | 写出一个可用 Skill | 设计一套可长期演进的 Skill 体系 |
| 关注点 | When / Procedure / Pitfalls | 架构、边界、复用、治理、版本化 |
| 适用人群 | 单人用户 | 重度用户、团队、知识库维护者 |
| 成功标准 | 当前任务可完成 | 长期稳定、低维护成本、高复用 |

### 什么时候你已经进入 Advanced Skills 阶段？

当你遇到以下任一情况时，就应该从“写 Skill”升级为“设计 Skill 体系”：
- 同一类 Skill 已经出现 3 个以上变体
- 不同 Skill 之间开始互相依赖
- 你需要给团队或仓库建立统一规则
- 你发现“维护 Skill”已经比“创建 Skill”更费时间

## 2. 高级 Skill 设计的 10 大原则

### 1. 边界先于实现
先定义这个 Skill 的边界、输入、输出、不要做的事，再开始写步骤。

### 2. 单 Skill 不解决一切
复杂任务应拆成多个协作 Skill，而不是一个超长“大杂烩 Skill”。

### 3. 组合优于堆叠
优先设计“可组合的几个小 Skill”，而不是“一个极长但难维护的万能 Skill”。

### 4. 生命周期意识
每个 Skill 都要考虑：创建、使用、Patch、版本升级、废弃。

### 5. 失败经验必须沉淀
Pitfalls 不是附属内容，而是 Skill 价值最高的部分之一。

### 6. 先验证，再固化
重复出现至少 2~3 次的成功工作流，才值得升格为高级 Skill。

### 7. 面向维护者写作
不仅要让“使用者”看懂，还要让“未来维护者”能快速 patch。

### 8. 明确依赖关系
写清楚它依赖哪些工具、哪些上下文文件、哪些其他 Skill。

### 9. 版本变化可追踪
重大改动要提升版本，并在变更记录中写清楚为什么变。

### 10. 始终可审查
高质量高级 Skill 必须支持系统性 Review：可评分、可比较、可替换。

### 好的高级 Skill 体系 vs 坏的高级 Skill 体系

| 维度 | 坏体系 | 好体系 |
|------|--------|--------|
| 命名 | 随意命名、用途不明 | 名称清晰、层级明确 |
| 边界 | 功能互相重叠 | 每个 Skill 有明确边界 |
| 维护 | 出问题后才修 | 有 Review 节奏与版本演进 |
| 复用 | 大量复制粘贴 | 小 Skill 可组合复用 |
| 治理 | 个人随意维护 | 有 Style Guide / Checklist / Maintainer |

## 3. Skill 架构模式

### 模式 1：单体型（Monolithic Skill）
适合低复杂度、单一任务。

**优点**：简单直接  
**缺点**：扩展性差、后期容易臃肿

### 模式 2：流水线型（Pipeline Skills）
将任务拆成多个顺序 Skill，例如：调研 → 总结 → 输出 → 发布。

**适用场景**：内容生产、自动化报告、研究流程

### 模式 3：协调器型（Orchestrator Skill）
一个高层 Skill 负责判断、拆分、调度多个子 Skill。

**适用场景**：复杂项目维护、仓库级自我进化、多人协作任务

### 模式 4：守门员型（Gatekeeper Skill）
专门负责质量审查、发布前检查、合规校验。

**适用场景**：代码提交前、文档发布前、知识库更新前

### 模式 5：维护者型（Maintainer Skill）
围绕某一知识库、项目、文档体系进行长期自动维护。

**代表案例**：`startbook-master-maintainer`

### 模式选择表

| 模式 | 复杂度 | 推荐场景 | 推荐度 |
|------|--------|----------|--------|
| 单体型 | 低 | 单一高频任务 | ★★★☆☆ |
| 流水线型 | 中 | 可拆分的固定流程 | ★★★★★ |
| 协调器型 | 高 | 多步骤复杂任务 | ★★★★★ |
| 守门员型 | 中 | 质量把关 | ★★★★☆ |
| 维护者型 | 高 | 长期知识库/仓库维护 | ★★★★★ |

### 架构选择建议

- 任务固定、边界清晰 → 单体型
- 步骤稳定、顺序明确 → 流水线型
- 子任务多、判断复杂 → 协调器型
- 输出必须经过检查 → 守门员型
- 目标是长期治理 → 维护者型

## 4. 何时拆分、组合或废弃 Skill

这是高级 Skill 设计中最关键、也最容易做错的一步。

### 拆分决策树

当出现以下情况时，应优先考虑拆分：
- 一个 Skill 的步骤超过 10 步，且包含多个不同目标
- Pitfalls 已经覆盖多个完全不同的问题域
- 同一个 Skill 需要不同工具集、不同上下文环境
- 维护者经常不确定“应该 patch 哪一段”

### 组合决策树

当出现以下情况时，应优先考虑组合：
- 两个 Skill 总是连续使用
- 两个 Skill 共享同一输入/输出结构
- 高层流程比单个 Skill 更值得沉淀

### 废弃决策树

当出现以下情况时，应考虑退役：
- 已被更高质量 Skill 完全替代
- 关联工具或工作流已失效
- 长期没有使用，且继续维护没有收益

### 拆分 vs 组合 vs 废弃 对比表

| 选择 | 触发信号 | 推荐动作 |
|------|----------|----------|
| 拆分 | 一个 Skill 过大、边界混乱 | 拆成多个职责单一的 Skill |
| 组合 | 多个 Skill 总是一起使用 | 新增协调器型或流水线型 Skill |
| 废弃 | 旧 Skill 不再带来价值 | 删除或标记 deprecated |

## 5. 组合式 Skill 设计

组合式设计的核心不是“多”，而是“职责清晰 + 接口明确”。

### 一个典型组合

- `project-kickoff`：初始化项目
- `docs-style-enforcer`：统一文档规范
- `startbook-master-maintainer`：长期维护整个知识库
- `release-checker`：发布前最终检查

### 组合设计 Checklist

- 每个 Skill 是否有清晰边界？
- 是否存在功能重叠？
- 高层协调逻辑是否应该单独抽成 Orchestrator Skill？
- 依赖关系是否可被新维护者快速理解？
- 是否有统一输入输出格式？

### 组合案例：文档知识库维护流

1. `docs-style-enforcer` 负责规则检查
2. `startbook-master-maintainer` 负责整体 Review 和升级
3. `release-checker` 负责发布前质量门禁
4. `weekly-evolution-reporter` 负责输出周期性报告

**最佳实践**：如果两个 Skill 总是一起使用，就应该思考是否需要引入一个“协调器型”高级 Skill。

## 6. Skill 生命周期管理

### 生命周期 5 阶段

1. **发现阶段**：某类任务重复出现
2. **抽象阶段**：总结稳定方法
3. **固化阶段**：创建 Skill v1.0
4. **演进阶段**：使用 `patch` 累积 Pitfalls 与优化步骤
5. **退役阶段**：被更好的 Skill 替代，或需求消失

### 生命周期表

| 阶段 | 核心动作 | 成功标志 |
|------|----------|----------|
| 发现 | 观察重复任务 | 已重复出现 2~3 次 |
| 抽象 | 提炼步骤与边界 | 能清晰说出何时用/何时不用 |
| 固化 | 创建 v1.0 | 有完整模板与验证方式 |
| 演进 | 不断 patch | Pitfalls 越来越真实具体 |
| 退役 | 删除或合并 | 不再维护低价值 Skill |

### 生命周期治理建议

- 每个高价值 Skill 至少每月 Review 一次
- 每次重大 patch 都应重新检查 Verification
- 当 Skill 增长过快时，优先考虑拆分而不是继续堆内容

## 7. 质量评审体系

### 高级 Skill 评分模型（100 分）

| 维度 | 分值 | 说明 |
|------|------|------|
| 边界清晰度 | 20 | 是否清楚说明适用与不适用场景 |
| 步骤可执行性 | 20 | 是否有具体命令、Prompt、工具说明 |
| Pitfalls 质量 | 20 | 是否记录真实失败经验 |
| 组合兼容性 | 15 | 是否容易与其他 Skill 协作 |
| 可维护性 | 15 | 是否方便 patch、versioning、review |
| 验证机制 | 10 | 是否有清晰的成功标准 |

### 判定标准

- **90~100**：生产级，可长期复用
- **75~89**：良好，可继续迭代
- **60~74**：可用，但需要补齐 Pitfalls 与验证
- **<60**：应重构或废弃

### Review 时的 7 个关键问题

1. 这个 Skill 是否仍然有价值？
2. 它的边界是否比上个版本更清晰？
3. 是否记录了最近真实踩到的坑？
4. 是否能与周边 Skill 更自然地协作？
5. 是否有明显重复内容应该抽离？
6. 版本号是否真实反映了变化幅度？
7. 维护者是否能在 5 分钟内理解它？

## 8. 生产级模板

### 模板 1：Master Maintainer 型

适用：维护一个文档仓库、知识库或长期项目。

核心组成：
- 先加载 Style Guide
- 先生成 Plan
- 逐文档 Review
- 精确 Patch / Rewrite
- 输出 Evolution Report

### 模板 2：Research Orchestrator 型

适用：复杂研究任务。

核心组成：
- 拆分为资料收集、观点对比、批判性分析、报告输出
- 需要 `delegate_task` 支持并行子研究
- 最后将高价值洞见写入 Memory

### 模板 3：Project Bootstrapper 型

适用：新项目启动。

核心组成：
- 创建目录结构
- 生成 README、SOUL.md、Context Files
- 初始化 Skill 和 Memory 策略
- 建立后续维护节奏

### 模板 4：Release Gatekeeper 型

适用：发布前质量校验。

核心组成：
- 检查结构完整性
- 检查规范符合度
- 检查链接、版本号、Checklist
- 给出 go / no-go 结论

### 模板 5：Skill Portfolio Manager 型

适用：管理大量 Skill 的个人或团队。

核心组成：
- 统计现有 Skill 分类
- 找出重复、低质量、过期 Skill
- 给出拆分/合并/退役建议
- 维护整体命名规则与版本状态

### 模板选型建议

| 模板 | 适用阶段 | 价值 |
|------|----------|------|
| Master Maintainer | 长期维护 | 稳定提升一致性 |
| Research Orchestrator | 复杂研究 | 提升并行与深度 |
| Project Bootstrapper | 项目起步 | 快速建立规范 |
| Release Gatekeeper | 发布前 | 降低错误发布风险 |
| Skill Portfolio Manager | Skill 数量上升后 | 控制复杂度与债务 |

## 9. 反模式与避坑

### 反模式 1：万能大 Skill
试图把研究、写作、发布、维护全部塞进一个 Skill。

**问题**：难维护、难 patch、边界模糊。

### 反模式 2：只有步骤，没有边界
Skill 写了很多步骤，但没有说什么时候不该使用它。

### 反模式 3：没有失败经验
没有 Pitfalls 的 Skill，通常只是“纸面流程”，不是真正可用的生产知识。

### 反模式 4：不做版本管理
长期 patch 却不更新版本，后期难以追踪质量变化。

### 反模式 5：不做废弃治理
旧 Skill 即使失效也一直保留，最终拖累维护成本。

### 反模式 6：命名体系混乱
同一类型 Skill 使用完全不同命名规则，团队协作时极易失控。

**💡 最佳实践**：高级 Skill 设计的目标不是“写更多”，而是“让系统更清晰、更稳、更容易演进”。

## 10. 团队与仓库级维护策略

如果你不是单人使用，而是团队级使用 Hermes，建议建立以下规范：

1. 统一 Style Guide
2. 统一 Skill 命名规则
3. 每周一次 Skill Review
4. 建立“废弃 Skill 清理机制”
5. 关键高级 Skill 必须附带评分与审查清单
6. 高价值 Skill 必须指定维护责任人

### 团队治理最小落地方案

| 事项 | 最低建议 |
|------|----------|
| 命名规范 | 全部使用 kebab-case |
| Review 周期 | 每周或每两周一次 |
| 版本管理 | 重大变化必须升级版本 |
| 归档机制 | deprecated Skill 移入单独目录 |
| 主维护者 | 至少 1 名负责治理体系 |

**推荐做法**：
- 基础篇文档教“如何写”
- 进阶篇文档教“如何治理”
- Master Maintainer 负责长期一致性
